# AradCore 后续开发任务

本文是无聊天上下文交接入口。开始开发前先阅读 `README.md`、`HANDOFF.md`、根目录 `装备系统设计文档.md`，再核对测试服实际插件版本。禁止直接把当前原型放入正式服。

## 当前状态

- 当前版本：`0.1.0-SNAPSHOT`，Java 21、Gradle Kotlin DSL、Paper API `1.21.5-R0.1-SNAPSHOT`。
- 已有：统一品级概率、品级 PDC、基础值 PDC、数值缩放、Lore、背包扫描、`inspect/roll/set/reload` 命令、MythicLib 反射桥。
- 目标运行环境记录为 Paper 26.2、MMOItems 6.10.1 开发版、MMOCore 1.13.1 开发版、MythicLib 1.7.1 开发版、CoreTools 0.4.3-SNAPSHOT；必须从测试服再次确认。
- 当前没有 Gradle Wrapper、没有已验证构建产物、没有自动化测试，也没有测试服验证报告。
- `README.md`、`HANDOFF.md` 和 `config.yml` 当前显示中文乱码；先判断是文件内容已损坏还是终端编码显示错误。修复前保留原文件并以 UTF-8 无 BOM 统一验证。
- 当前 PDC 协议：`aradcore:grade_percent`、`aradcore:grade_schema`、`aradcore:base_<normalized-tag>`、`aradcore:grade_lore`。不得随意重命名；协议变化必须提供迁移器。

## P0：建立可重复构建

1. 添加 Gradle Wrapper，并固定可用 Gradle 版本；缓存、临时下载放在 `E:\AI\Codex\cache` 或 `E:\AI\Codex\tmp`。
2. 将 Paper API 版本与实际 Paper 26.2 对齐。确认服务器使用的 Minecraft API 版本，不能仅沿用 `1.21.5` 猜测。
3. 优先将测试服的 MMOItems、MythicLib JAR 作为 `compileOnly` 本地依赖或接入其官方仓库，以便编译期验证 API；反射只能作为开发版兼容边界，不能替代契约测试。
4. 执行 `gradlew clean build`、单元测试和静态检查；生成物只能来自干净工作树。
5. 增加 CI（至少 Java 21 构建与测试），不把服务器插件 JAR、玩家数据、RCON 信息提交到仓库。
6. 修正资源编码后验证 `plugin.yml`、`config.yml` 能由 Bukkit YAML 正确加载，中文在控制台及游戏内均正常。

## P0：代码正确性与数据安全风险

以下问题解决前不得测试服批量处理物品，更不得上正式服：

1. **NBT 与 ItemMeta 覆盖顺序**：`GradeService.ensure/regrade` 先通过包装器写 NBT、调用 `toItem()`，随后把写入前取得的旧 `ItemMeta` 设置到结果上。现代物品组件/PDC/NBT 可能因此被旧 meta 覆盖。必须用真实 MythicLib API 验证并重构为单一写入事务：从同一最终物品读写 PDC、属性和 Lore，最后只生成一次结果。
2. **PDC 所在副本不明确**：基础值与品级先写入输入物品的 meta，NBT bridge 可能返回克隆。必须证明 PDC 最终存在于返回物品，重启后仍在。
3. **错误标签类型**：`getDouble/setDouble` 假设所有目标标签都是 double；真实标签可能为 int、float、字符串、列表或根本不在 MythicLib NBT 中。不得对未知类型写入。
4. **标签路径模型**：配置称其为“路径”，实现只接受扁平键。必须确认 MythicLib 版本的 NBTItem API、嵌套结构、组件化物品格式和 `getTags()` 返回语义。
5. **身份判断过宽/过窄**：仅凭任一 identity tag 判断 MMOItem，可能误处理或漏处理。优先使用 MMOItems 官方 API（类型、ID、是否有效 MMOItem），并支持模板/类型黑白名单。
6. **事件时序错误**：背包扫描发生在物品生成后，可能与 MMOItems lore builder、装备识别、掉落、CoreTools GUI、堆叠同步互相覆盖。优先接入 MMOItems `ItemBuildEvent` 的正确阶段，并明确生成原因；背包扫描只能作为迁移/修复工具。
7. **事件风暴和覆盖玩家背包**：每次点击/拖动均延迟扫描整个背包，多个任务不去重，`setContents` 可能覆盖同 tick 的其他插件修改。实现按玩家去抖、只处理受影响槽位、版本/指纹校验和冲突放弃策略。
8. **Join 配置逻辑缺陷**：`later()` 同时受 `on-inventory-events` 控制，导致 `on-join=true`、`on-inventory-events=false` 时 Join 扫描也可能不执行。配置开关应由各事件入口独立控制。
9. **Lore 不可靠**：以首行为管理行会覆盖 MMOItems 或其他插件的首行；PDC marker 不能定位具体 Component。应由 MMOItems lore builder/占位符渲染，或使用稳定 marker 与重新构建策略。测试 MiniMessage 动态颜色标签是否合法，并处理用户配置中的非法 MiniMessage。
10. **旧物品基值污染**：没有 `base_*` 的既有装备会把当前已随机/强化后的值当作模板基值。必须提供显式迁移模式，不得静默接管。
11. **幂等与公式分层**：品级、强化、增幅、锻造、宝石、词条必须各自保存源数据/层数据，最终属性由统一计算器重建，禁止互相读取“当前最终值”继续乘算。
12. **概率配置校验**：拒绝负权重、反转区间、越界区间、重叠策略不明确、总权重为零、整数溢出。启动时输出明确错误并停用模块。
13. **异常策略**：反射异常目前逐物品刷控制台。增加限频、指标、结构化审计；出现契约不匹配时熔断品级写入，只允许 inspect。
14. **并发与主线程**：所有 Bukkit 物品操作必须在主线程；未来数据库/经济操作需分离异步 IO 与主线程物品提交，并进行提交前版本校验。
15. **复制/堆叠漏洞**：明确带 PDC 装备不可堆叠规则，覆盖创造复制、铁砧、砂轮、锻造台、漏斗、死亡掉落、容器、交易、邮件/拍卖、跨服传输。

## 真实 MMOItems 标签校准流程

只在隔离测试服执行，先备份到 `E:\Server\Minecraft\Beta\Backup`：

1. 记录精确版本与 JAR 哈希：Paper、MMOItems、MythicLib、MMOCore、Nexo、CoreTools。
2. 为每种装备类型制作最小模板，每件只设置一个待测属性；再制作包含物攻、魔攻、独立攻击、物防、魔防、生命、魔法、力量、智力、体力、精神的组合模板。
3. 通过 MMOItems 官方命令/API 分别生成物品；在生成前、`ItemBuildEvent` 各优先级阶段、进入背包后运行 `/aradcore inspect`，保存脱敏输出。
4. 同时使用 MMOItems/MythicLib 官方 debug/API 读取：物品 type、ID、stat map、NBT/PDC/组件键及其 Java 类型。不要仅凭键名猜测。
5. 每个属性做 A/B：模板值 100，设置品级 50%，预期最终值 50；再设 80%、50%，确认始终从 100 重建。
6. 检查显示 Lore、装备后实际战斗属性、重新登录、重启服务器、容器转移后的值是否一致。显示正确但战斗值错误视为失败。
7. 确认哪些属性是 MMOItems 模板 stat、哪些由 MythicLib 动态计算、哪些来自原版 AttributeModifier；分别建立类型安全 adapter，不再依赖统一 `getDouble`。
8. 将已确认的键、类型、API 方法、事件阶段、样本 inspect 输出写入仓库文档和契约测试 fixture；未确认标签不得加入默认 `scaled-tags`。

## P0 测试矩阵

### 自动化测试

- 概率：边界 1/100、所有区间可达、非法配置拒绝、固定随机源下可复现。
- 幂等：同一物品重复 `ensure` 100 次，PDC、属性和 Lore 均不变化。
- 重调：100→50→80→50 后与首次 50 完全一致；负数、0、101、超大值按明确策略处理。
- 精度：正数、零、小数、大数、负属性的舍入规则固定；不得产生 NaN/Infinity。
- 原子性：模拟任意属性读取/写入失败，物品、货币、材料均保持原状。
- 序列化：物品保存/加载后 PDC、基值、schema、最终值一致。
- 迁移：schema 0/1/未知版本、缺少部分 base key、旧 Lore、已强化旧物品均有明确结果且不静默损坏。
- Lore：空 Lore、非空 Lore、第三方改序、重复 marker、非法模板、reload 后均安全。
- 身份：原版物品、MMOItem、Nexo 物品、带相似自定义键物品不会误判。

### 测试服集成测试

- 生成、拾取、背包点击/拖动、快捷栏换手、装备、死亡、掉落、容器、漏斗、交易、拍卖/邮件、铁砧、砂轮、锻造、创造复制。
- MMOItems reload、AradCore reload、服务器重启、插件禁用/启用后的数据一致性。
- 多插件同 tick 修改同一物品时不丢改动；快速连点不会创建大量扫描任务。
- 物品显示值、MMOCore 玩家面板值和实际对怪物造成/承受伤害一致。
- 10,000 次生成的分布符合配置容差，并记录耗时、主线程峰值与日志量。

## P1：CoreTools、经济事务与审计

1. 定义稳定服务 API：`inspect`、`preview`、`roll`、`apply`、`canApply`、`rollback`；CoreTools 只负责 GUI，不直接改 NBT/PDC。
2. GUI 打开时生成物品快照/指纹；确认时重新验证玩家、槽位、物品 UUID/指纹、schema、余额和材料，防止换物骗取。
3. 建立事务顺序：预检 → 锁定物品 → 计算预览 → 扣除经济/材料 → 写入新物品 → 提交审计。任一步失败必须补偿退款；重试必须幂等。
4. 通过 Vault 或服务器明确经济实现建立适配器，禁止硬编码控制台命令扣款。定义金币不足、离线、数据库超时、退款失败处理。
5. 每件装备增加稳定实例 ID；审计记录操作者、来源/目标品级、材料、金额、时间、服务器、结果、错误、事务 ID，但不得记录密码或无关个人数据。
6. 提供管理员查询、按事务回滚、重复交易检测、审计保留/轮转策略。日志放插件正式数据目录；开发临时日志放 `E:\AI\Codex\logs`。
7. 权限、冷却、GUI 重复确认、防连点、断线恢复、崩溃恢复全部纳入测试。

## P2：装备成长路线

按统一计算引擎逐模块开发，每个模块完成测试后再进入下一模块：

1. **强化**：等级、成功率、金币/材料、失败降级/保护、武器与防具成长曲线、继承规则。
2. **增幅**：异次元属性、增幅等级、与强化互斥或转换规则、保护券和失败结果。
3. **锻造**：独立攻击或职业特定成长，明确上限、材料和是否失败。
4. **耐久与修理**：最大/当前耐久、损耗事件、不可破坏场景、普通/完全修理、修理成本。
5. **镶嵌**：孔位数量/颜色/类型、开孔、宝石装卸、绑定与返还、重复唯一效果约束。
6. **词条与洗练**：词条池、权重、等级、冲突组、锁定词条成本、保底与预览。
7. **继承/铭刻/转移**：允许转移的层、来源销毁策略、跨类型限制、回滚审计。
8. **套装与装备需求**：职业、等级、品质、绑定、套装计数；通过统一 API 暴露给 MMOCore/PAPI。

建议最终公式明确为纯函数并版本化：模板基础值 × 品级系数 + 强化层 + 增幅/锻造层 + 宝石/词条层；所有显示和战斗消费者读取同一计算结果。

## 完成定义（Definition of Done）

### 0.1 品级模块可交付

- Gradle Wrapper 与 CI 可在干净环境用 Java 21成功构建；测试全绿并产出可追溯 JAR。
- 已针对实际 MMOItems/MythicLib 版本完成标签/API/事件契约校准，默认配置只包含已验证属性。
- 上述 P0 数据覆盖、事件时序、Lore、迁移、幂等、异常熔断问题全部解决并有测试。
- 在隔离测试服完成测试矩阵；无物品复制、属性叠乘、数据丢失、背包覆盖或控制台刷屏。
- 有备份、schema migration、关闭模块后的行为、回滚步骤和测试报告。
- 代码审查确认未提交服务器机密、玩家数据、付费插件 JAR 或 RCON 密码。

### 整体 AradCore 可交付

- CoreTools 只通过稳定 API 调用；经济和物品变更原子、可回滚、可审计。
- 强化、增幅、锻造、修理、镶嵌等模块均使用统一版本化计算器，没有重复乘算或执行顺序差异。
- 管理命令、权限、中文消息、PAPI、迁移工具、运维文档、性能基准齐全。
- 测试服持续运行并覆盖重启、崩溃恢复和真实多人操作后，由服主明确批准才进入正式服。
